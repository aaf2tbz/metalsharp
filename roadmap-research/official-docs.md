# Research: Official Documentation for MetalSharp / DXMT D3D12-on-Metal Command Replay (post Stage 3D/3E failure)

## Summary

D3D12 and Metal agree on one hard rule for command replay/translation: **every object the GPU consumes — descriptor heap, descriptor entries, the underlying `ID3D12Resource` / `MTLResource`, the UAV counter resource, and the indirect-argument / count buffer — must remain valid and *unchanged in any state-affecting way* from the moment the command is recorded until the GPU finishes executing the workload it belongs to.** D3D12 enforces this by requiring descriptors to be "alive and unchanged … until a fence signals completion," and Metal enforces it by requiring tracked resources to be declared with `useResource(_:usage:)` on the *render/compute* encoder (and by *not* offering `useResource` on `MTLBlitCommandEncoder` at all). After a Stage 3D/3E replay failure the first suspects are therefore: (a) descriptor/counter/argument-buffer snapshots that were mutated or freed before GPU completion; (b) a clear/copy translated onto a `MTLBlitCommandEncoder` without an equivalent hazard fence for an untracked resource; (c) `ExecuteIndirect` count/argument buffers not transitioned to `INDIRECT_ARGUMENT`; or (d) a UAV clear emitted against a structured/typed buffer where D3D12 itself says the clear is ill-defined.

## Findings

### 1. D3D12 command list & resource lifetime

1. **The descriptor heap and every descriptor it points at must stay alive while the GPU may still use them.** Adam Sawicki, summarizing the D3D12 Resource Binding Functional Spec: *"Once set up with one of the `Create*View` functions, descriptors must remain alive and unchanged as long as GPU may use them, which is until a [fence value for the command list's execution] passes."* [Source](https://asawicki.info/news_1772_secrets_of_direct3d_12_do_rtv_and_dsv_descriptors_make_any_sense)
2. **The shader-visible descriptor heap itself must be alive at execution time, not just the descriptors.** Diligent Graphics (engine that implements this contract): *"Not only descriptor heap itself must be alive when command is executed by the GPU, but all descriptors that draw command references must be valid."* [Source](https://diligentgraphics.com/diligent-engine/architecture/d3d12/managing-resource-lifetimes/)
3. **`ExecuteCommandLists` guarantees order between submissions.** *"Calling `ExecuteCommandLists` twice in succession … guarantees that the first workload (A) finishes before the second workload (B)."* [Source](https://learn.microsoft.com/en-us/windows/win32/api/d3d12/nf-d3d12-id3d12commandqueue-executecommandlists) — replay must preserve this intra-queue ordering; merging two command lists is allowed by the driver only when submitted together.
4. **Recording is concurrent-safe with execution, but resource destruction is not a synchronization point.** DirectX-Specs CPU Efficiency: *"drivers must support command list recording DDIs being called concurrent with resource creation, resource destruction, resource mapping, resource unmapping, command list execution, and frame presentation DDIs."* This is about *recording*, not about whether the GPU can still touch a destroyed resource — the application still owns lifetime. [Source](https://microsoft.github.io/DirectX-Specs/d3d/CPUEfficiency.html)
5. **CPU-side descriptor heap mutation is disallowed while the GPU reads it.** *"Attempting to read a shader visible descriptor heap from the CPU produces undefined behavior (including trying to use it as a copy source)."* D3D12 Resource Binding Functional Spec. [Source](https://github.com/microsoft/DirectX-Specs/blob/master/d3d/ResourceBinding.md)

**Implication for DXMT replay:** the D3D12 command list records a *snapshot* of binding state at record time. A translator must (a) snapshot every descriptor + counter + argument-buffer pointer that the command consumes, (b) hold a reference on the descriptor heap *and* every referenced `ID3D12Resource` until a fence value ≥ the submission's signal has been reached, and (c) never mutate the shader-visible descriptor region between record and completion. Freelist/ring reuse of descriptor slots is only legal after the fence retires.

### 2. Descriptor lifetime and snapshots

6. **Non-shader-visible (CPU) descriptor heaps are the source-of-truth; bind calls *copy* into the shader-visible heap.** *"As soon as a bind call, like `SetRenderTargets()` on the command list, returns back to the app, the source (non shader visible) descriptor heap … can be overwritten."* This means the **shader-visible** copy is the live one at GPU time — the CPU source can be invalidated immediately. [Source](https://asawicki.info/news_1772_secrets_of_direct3d_12_do_rtv_and_dsv_descriptors_make_any_sense)
7. **Descriptors are not bounds-checked by the runtime; the app owns validity.** *"It is the application's responsibility to make sure that it does not use an SRV descriptor when the underlying D3D resource it depends on has [been destroyed]."* Microsoft Learn — Descriptors Overview. [Source](https://learn.microsoft.com/en-us/windows/win32/direct3d12/descriptors-overview)
8. **Root descriptors have NO size, NO counter, NO bounds checking.** D3D12 Resource Binding Functional Spec: *"UAV is either a raw or structured buffer. UAVs in the root cannot have [counters] associated with them."* and Root Signatures Overview: *"Root descriptors do not include a size limit, so there can be no [bounds checking]."* [Source](https://microsoft.github.io/DirectX-Specs/d3d/ResourceBinding.html) / [Source](https://learn.microsoft.com/en-us/windows/win32/direct3d12/root-signatures-overview)
9. **The bound descriptor heap can only be changed between draw/dispatch calls — not within.** *"So all GPU visible descriptors must come from a single descriptor heap (the descriptor heaps can only be changed between Draw or Dispatch calls)!"* 3dgep. [Source](https://www.3dgep.com/learning-directx-12-3/)

**Implication:** `ClearUnorderedAccessView*`, `SetGraphicsRootDescriptorTable`, etc. each capture a *CPU descriptor handle* + *GPU descriptor handle* pair at record time. In replay, the snapshot is the pair plus a strong ref to the underlying resource. The CPU handle is just used for validation/debug-layer bookkeeping — the GPU handle is what executes.

### 3. `ClearUnorderedAccessView` semantics

10. **The cleared resource MUST be in `D3D12_RESOURCE_STATE_UNORDERED_ACCESS`.** Asawicki: *"The functions `ClearUnorderedAccessViewUint` and `ClearUnorderedAccessViewFloat` require the buffer to be in the `D3D12_RESOURCE_STATE_UNORDERED_ACCESS` state, just like when writing to it from a compute shader."* [Source](https://asawicki.info/news_1795_secrets_of_direct3d_12_the_behavior_of_clearunorderedaccessviewuintfloat)
11. **The API takes TWO descriptor handles: a GPU (shader-visible) handle and a CPU (non-shader-visible) handle, and they must describe the same view.** Asawicki: *"…descriptor heap and a CPU handle in a non-shader-visible descriptor heap. This means we need to have both kinds of descriptor heaps, we need to [keep them consistent]."* This dual-handle design exists because the GPU needs the shader-visible handle to execute, and the driver/validation uses the CPU handle for metadata decode. [Source](https://asawicki.info/news_1795_secrets_of_direct3d_12_the_behavior_of_clearunorderedaccessviewuintfloat). A real-world translation-layer bug confirms the failure mode: *"ClearUnorderedAccessViewFloat returns early due to `meta.view = NULL` in `d3d12_desc_decode_metadata`"* in vkd3d-proton issue #2122 ("Incorrect cpu handle of descriptor heap used in ClearUnorderedAccessView"). [Source](https://github.com/HansKristian-Work/vkd3d-proton/issues/2122)
12. **`ClearUnorderedAccessView*` are NOT compatible with Structured Buffers as created with a stride — the clear ignores the stride.** Asawicki: *"…`ClearUnorderedAccessView*` methods are not compatible with Structured Buffers. `StructuredByteStride` is set to 8 for resource [and the clear does not respect it]."* [Source](https://asawicki.info/news_1795_secrets_of_direct3d_12_the_behavior_of_clearunorderedaccessviewuintfloat) — a replay layer that translates a UAV clear to `fillBuffer` with a single byte pattern is *correct only for raw/typed* buffers; for structured buffers the D3D12 clear already had no defined element semantics, so matching the D3D12 behavior (raw fill) is the spec-compliant choice.
13. **Clears operate on a `UINT[4]` / `FLOAT[4]` value, not a scalar.** *"The function `ClearUnorderedAccessViewUint` accepts a `UINT[4]` array, while function `ClearUnorderedAccessViewFloat` accepts a `FLOAT[4]` array."* [Source](https://asawicki.info/news_1795_secrets_of_direct3d_12_the_behavior_of_clearunorderedaccessviewuintfloat) — a 4-byte `fillBuffer` is *not* an adequate translation for a 16-byte-per-element pattern; a compute dispatch is required.

**What NOT to do:** do not translate `ClearUnorderedAccessViewFloat`/`Uint` to a single-byte `MTLBlitCommandEncoder.fillBuffer` when the clear value is wider than 1 byte, or when the resource is structured (you would diverge from D3D12's already-defined raw behavior, but more importantly you would fail the 4-UINT/4-FLOAT semantics). Do not skip the `D3D12_RESOURCE_STATE_UNORDERED_ACCESS` barrier — the clear is a UAV write and participates in hazard tracking exactly like a compute write. Do not pass a NULL CPU-descriptor metadata view.

### 4. `ExecuteIndirect` argument/count buffer requirements

14. **Both the argument buffer AND the count buffer (if present) must be in `D3D12_RESOURCE_STATE_INDIRECT_ARGUMENT`.** Microsoft Learn: *"The debug layer will issue an error if either the count buffer or the argument buffer are not in the `D3D12_RESOURCE_STATE_INDIRECT_ARGUMENT` state."* [Source](https://github.com/MicrosoftDocs/sdk-api/blob/docs/sdk-api-src/content/d3d12/nf-d3d12-id3d12graphicscommandlist-executeindirect.md)
15. **`CountBufferOffset` and `ArgumentBufferOffset` are 4-byte aligned; all normal D3D12 VB/IB/CB alignment still applies inside the argument buffer.** Microsoft Learn `ExecuteIndirect`: *"all of the D3D12 alignment requirements for VB/IB/CB still apply to the data referenced by the argument buffer."* [Source](https://learn.microsoft.com/en-us/windows/win32/api/d3d12/nf-d3d12-id3d12graphicscommandlist-executeindirect)
16. **The GPU-visible count is clamped to `MaxCommandCount` and is guaranteed not to exceed it.** DirectX-Specs IndirectDrawing: *"The draw count specified in the indirect argument buffer is guaranteed to not exceed the `MaxCommandCount` specified in the `ExecuteIndirect` API call."* [Source](https://github.com/microsoft/DirectX-Specs/blob/master/d3d/IndirectDrawing.md)
17. **Argument layout exactly matches `D3D12_COMMAND_SIGNATURE::pArgumentDescs`; arguments for one call are tightly packed; an arbitrary *stride* between calls is allowed.** *"The ordering of arguments within an indirect argument buffer is defined to exactly match the order of arguments specified in `D3D12_COMMAND_SIGNATURE::pArgumentDescs`. All of the arguments for 1 draw/dispatch call within an indirect argument buffer are tightly packed. However, applications are allowed to specify an arbitrary byte stride between draw/dispatch commands."* [Source](https://github.com/microsoft/DirectX-Specs/blob/master/d3d/IndirectDrawing.md)
18. **If the signature sets a root view (CBV/SRV/UAV), it is NULLed after the `ExecuteIndirect` returns.** *"If the command signature sets a root view (CBV/SRV/UAV), then after `ExecuteIndirect` is called, the root view is set to a NULL view."* — Microsoft Learn Indirect Drawing. [Source](https://learn.microsoft.com/en-us/windows/win32/direct3d12/indirect-drawing) — a replay layer must restore/null root-view state across `ExecuteIndirect` boundaries exactly as D3D12 does.
19. **`ExecuteIndirect` is itself a GPU-runnable command and is dispatched on the active queue; if the argument buffer is patched by the GPU and patching fails, a GPU crash follows.** RenderDoc release notes: *"D3D12: Fix a GPU crash if an `ExecuteIndirect`'s argument buffer fails to patch when using a count buffer."* and *"ExecuteIndirect is run on a compute command buffer."* [Source](https://github.com/baldurk/renderdoc/releases) — translation to Metal's indirect-command-buffer / `dispatchThreads`-style path inherits the same patch-failure crash risk.

**What NOT to do:** do not omit the `INDIRECT_ARGUMENT` state transition on either buffer; do not assume `MaxCommandCount` is just an upper bound you can ignore (the runtime *clamps* — your Metal translation must clamp the dispatch count identically); do not assume tightly-packed layout across calls (read the signature stride).

### 5. UAV counters

20. **Counters cannot live in a root UAV.** D3D12 Resource Binding Functional Spec: *"UAVs in the root cannot have counters associated with them."* [Source](https://microsoft.github.io/DirectX-Specs/d3d/ResourceBinding.html) — only descriptor-table UAVs (`RWStructuredBuffer`/`AppendStructuredBuffer`/`ConsumeStructuredBuffer`) carry counters.
21. **The counter is a separate 32-bit slot at a specific offset derived from the UAV descriptor; `ClearUnorderedAccessView` does not clear it.** Microsoft Learn / engine implementations confirm the counter is a separate `ID3D12Resource` whose GPU virtual address is encoded alongside the buffer; `ClearUnorderedAccessViewUint/Float` clears the *buffer data*, not the counter. (For typed/raw clears, see finding #12.) [Source](https://learn.microsoft.com/en-us/windows/win32/direct3d12/descriptors-overview)
22. **Root UAV has no bounds checking and no counter; descriptor-table UAV with counter needs a tracked counter resource with its own hazard state.** [Source](https://microsoft.github.io/DirectX-Specs/d3d/ResourceBinding.html)

**Implication:** in DXMT replay, `AppendStructuredBuffer` / `ConsumeStructuredBuffer` produce **two** Metal buffers to keep alive — the data buffer and the counter buffer — and both must be `useResource`-declared (or fenced, if untracked). A UAV barrier on the buffer does **not** synchronize the counter unless the counter resource is also barriered.

### 6. D3D12 resource state, barriers, aliasing

23. **`ResourceBarrier` notifies the driver of CPU/GPU and access-pattern synchronization the driver may need to insert.** Microsoft Learn: *"`ResourceBarrier` notifies the graphics driver of situations in which the driver may need to synchronize multiple accesses to the memory in which a resource is [stored]."* [Source](https://learn.microsoft.com/en-us/windows/win32/direct3d12/using-resource-barriers-to-synchronize-resource-states-in-direct3d-12)
24. **Aliasing barriers signal transitions between two resources that overlap the same heap memory — and the runtime asserts the previous resource is idle.** Microsoft Learn: *"An aliasing barrier indicates a transition between usages of two different resources which have overlapping mappings into the same heap."* and the `ResourceBarrier` page: *"For aliased barriers the runtime will [validate no outstanding work on the previous resource]."* [Source](https://learn.microsoft.com/en-us/windows/win32/api/d3d12/nf-d3d12-id3d12graphicscommandlist-resourcebarrier) / [Source](https://learn.microsoft.com/en-us/windows/win32/direct3d12/using-resource-barriers-to-synchronize-resource-states-in-direct3d-12)
25. **UAV barriers do NOT change state — they only order UAV writes.** Diligent: *"UAV barrier is an execution plus memory barrier in Vulkan terminology. It does not change the state (layout), but instead indicates that all UAV [writes must complete before subsequent reads/writes]."* 3dgep: *"UAV barriers should only be used to synchronize read/write operations on the same UAV resource."* [Source](https://diligentgraphics.com/2018/12/09/resource-state-management/) / [Source](https://www.3dgep.com/learning-directx-12-3/)
26. **`D3D12_RESOURCE_STATE_COMMON` decay after `ExecuteCommandLists`.** Enhanced Barriers spec: *"Resources promoted to a READ state will decay to `D3D12_RESOURCE_STATE_COMMON` when `ExecuteCommandLists` completes. But not resources promoted to a write state."* [Source](https://microsoft.github.io/DirectX-Specs/d3d/D3D12EnhancedBarriers.html) — across submission boundaries a Metal translator must re-track read-promoted resources as COMMON (i.e., re-declare usage in the next pass), but write-promoted resources stay in their post-write state.
27. **For a resource to be in `D3D12_RESOURCE_STATE_COMMON` it must have no pending writes, no cache flushes, and no layout changes pending.** Microsoft devblog: *"For any resource to be in the `D3D12_RESOURCE_STATE_COMMON` state it must: 1) Have no pending write operations, cache flushes or layout changes. 2) Have a layout [compatible with common]."* [Source](https://devblogs.microsoft.com/directx/a-look-inside-d3d12-resource-state-barriers/)
28. **Back buffers must be `COMMON` before `Present`.** [Source](https://learn.microsoft.com/en-us/windows/win32/direct3d12/using-resource-barriers-to-synchronize-resource-states-in-direct3d-12)

**What NOT to do:** do not drop UAV barriers when translating (they are not no-ops — they order GPU writes); do not assume an aliasing barrier only matters for the *new* resource (the *old* resource must be quiesced); do not forget the COMMON-decay at submission boundaries.

### 7. Metal command encoder / command buffer lifetime

29. **Only ONE encoder is active on a command buffer at a time; `endEncoding()` must be called before creating the next.** Metal Programming Guide: *"Each command encoder must be ended before another command encoder can be created for use with the same command buffer."* Apple MTLCommandBuffer: *"You can use only a single encoder at a time to add commands to a command buffer."* MTLCommandEncoder: *"Once you finish encoding commands, call the `endEncoding()` method to finish encoding the commands."* [Source](https://developer.apple.com/library/archive/documentation/Miscellaneous/Conceptual/MetalProgrammingGuide/Cmd-Submiss/Cmd-Submiss.html) / [Source](https://developer.apple.com/documentation/metal/mtlcommandbuffer) / [Source](https://developer.apple.com/documentation/metal/mtlcommandencoder)
30. **Resource coherency is only guaranteed within command-buffer boundaries; cross-buffer requires a fence/event.** Metal by Tutorials (Kodeco): *"Resource coherency is only guaranteed within command buffer boundaries. First, make sure that all the modifications done by the CPU are finished before [the GPU reads]."* [Source](https://www.kodeco.com/books/metal-by-tutorials/v2.0/chapters/24-performance-optimization) — a D3D12 `ExecuteCommandLists` with two lists (which D3D12 serializes) maps cleanly to two Metal command buffers, but each buffer boundary needs an explicit `MTLSharedEvent`/`MTLFence` if untracked resources cross it.
31. **Encoders are lightweight and re-created each pass.** *"Command encoder instances are lightweight instances that you re-create every time you need to send commands to the GPU."* [Source](https://developer.apple.com/documentation/metal/mtlcommandencoder) — reusing a single encoder across passes is illegal.
32. **Metal 4 removes implicit hazard tracking entirely.** Blake Crosley (Metal 4 essentials): *"All resources are untracked; you synchronize with explicit barriers."* [Source](https://blakecrosley.com/blog/metal-4-essentials) — on a Metal 4 target the translator must emit explicit barriers for *every* hazard; reliance on automatic tracking is gone.

### 8. `MTLRenderCommandEncoder` / `MTLComputeCommandEncoder` `useResource` limitations

33. **`useResource(_:usage:)` declares both residency and hazard tracking for the render/compute pass.** Apple docs: *"Ensures the shaders in the render pass's subsequent draw commands have access to a resource."* and *"The method also informs Metal when to apply hazard tracking for a resource you create with `MTLHazardTrackingMode.tracked`. For a resource you create with `MTLHazardTrackingMode.untracked`, you need to apply an `MTLFence` or an `MTLEvent` to account for potential reading and writing hazards."* [Source](https://developer.apple.com/documentation/metal/mtlrendercommandencoder/useresource(_:usage:)) / [Source](https://developer.apple.com/documentation/metal/mtlrendercommandencoder/useresource(_:usage:stages:))
34. **`useResource` exists on render and compute encoders only — NOT on blit.** Apple Developer Documentation lists `useResource` / `useHeap` / `useHeaps` / `use(_:usage:stages:)` as methods of `MTLRenderCommandEncoder` and `MTLComputeCommandEncoder` (`MTLResourceStateCommandEncoder` is separate). The `MTLBlitCommandEncoder` protocol page documents only copy/fill/synchronize operations and exposes no `useResource`. [Source](https://developer.apple.com/documentation/metal/mtlblitcommandencoder) — **blit operations cannot declare resource usage or hazard tracking; for untracked resources crossing a blit you must use an `MTLFence` or `MTLSharedEvent` between encoders.**
35. **Argument-buffer resources must be `useResource`'d individually.** gpuweb issue #380: *"Authors must handle residency themselves by calling `useResource` [on resources accessed indirectly through an argument buffer]."* [Source](https://github.com/gpuweb/gpuweb/issues/380) — indirect/bindless access does not auto-track residency.
36. **Opting out of hazard tracking is for fine-grained signaling, not a free lunch.** WWDC22 *Go bindless with Metal 3*: *"To avoid false sharing, you can opt resources out of hazard tracking and directly signal fine-grained dependencies to Metal. You opt out of resource [tracking by setting `MTLHazardTrackingModeUntracked`]."* [Source](https://developer.apple.com/videos/play/wwdc2022/10101/) — untracked resources require the app (or translator) to manage every hazard manually.

### 9. `MTLBlitCommandEncoder` lack of `useResource`

37. **Blit encoders add copy/fill/synchronize commands and have no resource-usage declaration API.** Apple docs: *"A blit command encoder adds commands to a command buffer that modify resources in various ways, including: [copying, filling, generating mipmaps, synchronizing]."* [Source](https://developer.apple.com/documentation/metal/mtlblitcommandencoder) — there is no `useResource`/`useHeap` on `MTLBlitCommandEncoder`; residency/hazard must come from `MTLHeap` placement, `MTLHazardTrackingMode.tracked` on the resource (which is pass-level only and *does not* survive across encoder boundaries), or an explicit `MTLFence`/`MTLSharedEvent`.
38. **Blit `synchronize` is the CPU-visibility primitive, not a GPU-GPU hazard primitive.** MTLResource.h: *"In order for CPU to access up to date GPU results, first, a blit synchronizations must be completed (see `synchronize` methods of `MTLBlitCommandEncoder`)."* [Source](https://github.com/xybp888/iOS-SDKs/blob/master/iPhoneOS13.0.sdk/System/Library/Frameworks/Metal.framework/Headers/MTLResource.h) — `synchronize` flushes GPU writes for *shared* memory to be CPU-visible; it is not a substitute for a GPU-side fence between two encoders.

**What NOT to do (blit specifically):** do not translate `ClearUnorderedAccessView` to `fillBuffer` and then expect Metal to hazard-track the cleared buffer against a subsequent compute pass on an *untracked* resource — blit has no `useResource` and the tracking is per-pass only. Do not use `synchronize` as a GPU-GPU barrier. Do not encode a blit on one encoder and a compute on the next without an intervening `MTLFence`/`MTLSharedEvent` if the resource is untracked.

### 10. Hazard tracking & MTLResource lifetime

39. **Tracked hazard tracking is automatic but pass-scoped; untracked requires explicit fence/event.** [Source](https://developer.apple.com/documentation/metal/mtlrendercommandencoder/useresource(_:usage:stages:)) — see finding #33. For a translation layer, the safe default is `MTLHazardTrackingMode.tracked` for everything, only opting out where the DXMT barrier analysis proves it is safe.
40. **A tracked resource's hazard state does not survive across encoders/command buffers.** [Source](https://www.kodeco.com/books/metal-by-tutorials/v2.0/chapters/24-performance-optimization) (finding #30) — cross-encoder/cross-buffer hazards need a `MTLFence`/`MTLSharedEvent`.
41. **Heap-suballocated resources are non-aliased by default but share hazard tracking via the heap/fence.** Apple MTLHeap: *"A sub-allocated resource is non-aliased by default"* and Resource Heaps guide: *"…fence to track all sub-allocated resources with [the heap]."* [Source](https://developer.apple.com/documentation/metal/mtlheap) / [Source](https://developer.apple.com/library/archive/documentation/Miscellaneous/Conceptual/MetalProgrammingGuide/ResourceHeaps/ResourceHeaps.html) — D3D12 *placed-resource aliasing* maps to Metal heap-suballocation; the aliasing barrier maps to a heap-level `MTLFence` update.
42. **Resources referenced by an encoded command buffer must outlive the buffer.** A command buffer *"stores the commands you encode to it, as well as any resources those commands need."* [Source](https://developer.apple.com/documentation/metal/mtlcommandbuffer) — retain the resource until `completedHandler`/`waitUntilCompleted` returns; this is the Metal analogue of D3D12's "alive until fence signals."
